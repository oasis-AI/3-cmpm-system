## 摘 要

在运营商会员体系数字化转型持续推进的背景下，积分商城已成为连接用户运营与商品供给的重要平台。本文围绕中国移动积分商城场景，采用需求分析、系统设计、工程实现与功能测试相结合的方法，设计并实现了一个面向用户、商户和管理员三类角色的积分商城系统。系统采用前后端分离架构，后端基于 FastAPI 与 MySQL 实现业务服务，前端基于 Vue 3 构建交互界面，并引入 Redis 作为运行支撑组件。系统实现了注册赠积分、每日签到、积分兑换下单、快捷充值、订单跟踪、退款审核、商品与库存维护等功能。测试结果表明，系统能够完成积分获取、积分消费与订单处理的主要业务闭环，具有一定的工程实践价值和教学参考意义。

关键词：中国移动积分商城  积分商城系统  前后端分离  FastAPI  Vue 3

## ABSTRACT

With the continuous digital transformation of operator membership systems, the points mall has become an important platform connecting user operation and product supply. Focusing on the business scenario of China Mobile Points Mall, this paper combines requirement analysis, system design, engineering implementation and functional testing to design and implement a points mall system for users, merchants and administrators. The system adopts a front-end and back-end separated architecture. The back end is implemented with FastAPI and MySQL, the front end is built with Vue 3, and Redis is introduced as a runtime support component. Core functions include registration bonus points, daily check-in, points-based ordering, quick recharge, order tracking, refund review, and product and inventory maintenance. Test results show that the system can complete the main business loop of points acquisition, points consumption and order processing, and therefore has certain engineering and teaching value.

Keywords: China Mobile Points Mall; Points Mall System; Front-end and Back-end Separation; FastAPI; Vue 3

# 绪 论

伴随移动互联网、平台经济与用户精细化运营模式的发展，积分体系已从传统促销活动中的附属手段转变为企业数字资产运营的重要组成部分。对于通信运营商而言，积分具有双重属性：一方面，它承担用户活跃度提升、客户忠诚度维系和品牌触达的营销职能；另一方面，它又直接连接商品兑换、活动参与、权益交付与服务反馈，决定着用户对平台价值的直观感知。若积分只能停留在“发放”层面，而缺少清晰的消费路径、可管理的商品供给和可调控的运营规则，则难以形成高质量的运营闭环。

中国移动积分商城正是在这一业务诉求下形成的典型场景。与普通电商平台相比，积分商城的核心矛盾并不只是商品交易，而是积分资产流转、规则配置、订单履约和平台治理之间的协同问题。其系统设计不仅要满足前台商品展示与兑换体验，还要兼顾商户端的供给管理能力、管理员端的审核与运营能力，并在库存、积分与订单三类关键数据之间建立稳定的约束关系。因此，积分商城系统本质上是一个包含多角色协作、多状态流转和多维约束控制的综合型 Web 应用。

基于上述认识，本文以中国移动积分商城为研究对象，围绕“业务闭环完整、角色边界清晰、关键数据一致、平台具备可持续扩展能力”这一目标开展设计与实现工作。研究过程中综合采用文献梳理、需求分析、原型抽象、系统分层设计与工程实现验证等方法，构建了面向用户、商户和管理员三类主体的积分商城平台。系统既覆盖注册登录、积分发放、商品浏览、订单处理和售后退款等基础业务，也实现了商户审核、积分规则配置、公告管理、活动运营和平台统计等管理功能，从而形成较为完整的运营支撑体系。

从论文组织上看，第一章分析课题背景、研究现状、研究意义与主要内容；第二章阐述系统采用的核心技术及其适用性；第三章从可行性和需求两个维度明确系统建设边界；第四章给出总体架构、功能模块、关键流程与数据库设计；第五章说明系统各核心模块的落地方案；第六章结合测试环境和典型业务流程对系统进行验证；最后对全文工作进行总结，并讨论系统后续可完善的方向。

# 第一章 系统概述

## 1.1 研究背景

近年来，会员积分体系已广泛应用于通信、银行、零售、电商与生活服务等行业。企业通过积分将用户行为与平台激励机制连接起来，使积分成为促进留存、驱动转化和增强用户黏性的常见工具。尤其是在通信行业中，用户规模庞大、业务触点分散、运营活动频繁，积分商城不仅承担权益兑现的作用，还需要连接终端商品、话费流量、活动营销和服务评价等多种业务场景，这对系统的稳定性、灵活性与可配置性提出了更高要求。

然而，从实际应用情况看，传统积分系统普遍存在若干短板。其一，用户侧体验往往围绕“积分查询”展开，而对商品浏览、兑换决策、订单追踪和售后服务支持不足，导致积分价值感知弱化。其二，商户侧缺乏统一的商品、库存和订单协同机制，平台难以形成稳定的供给能力。其三，运营管理能力不足，积分规则、活动审核和公告配置常常依赖人工处理，效率较低。其四，在高频兑换场景下，如果积分扣减、订单生成和库存更新之间缺少严谨的约束机制，容易引发超卖、重复扣分或状态不一致等问题。

在此背景下，设计并实现一套兼顾业务真实性与工程可行性的积分商城系统，既具有现实应用价值，也具有较强的教学研究意义。该系统应当能够覆盖积分获取、积分消费、订单履约、售后处理和平台治理等关键链路，并通过合理的架构设计与数据模型支撑三类角色的协同工作。

## 1.2 研究现状

### 1.2.1 国外研究现状

国外关于积分体系与用户忠诚度平台的研究起步较早，应用场景主要集中在航空里程、连锁酒店、零售会员和在线订阅服务等领域。相关研究普遍强调积分机制与用户画像、消费偏好、推荐模型及营销策略之间的协同关系，积分已从单一奖励媒介发展为数据驱动运营的重要变量[1][2]。从业务模式上看，国外成熟平台更关注会员生命周期管理、个性化权益推荐和跨场景价值转化，体现出较强的运营精细化趋势。

在系统实现层面，国外研究更强调大规模平台环境下的弹性扩展、服务解耦和高并发处理能力。一些研究围绕云边协同部署、流式数据处理和微服务化治理展开，对交易一致性、资源弹性调度以及架构自适应调整提出了系统化方案[3][15]。这类研究为积分平台的高可用设计提供了工程启发，但其实现复杂度较高，通常面向超大规模商业系统，对教学型项目而言存在架构过重、开发成本高和部署要求复杂等问题。

### 1.2.2 国内研究现状

国内积分商城系统的建设主要分布于通信运营、银行信用卡、电商平台和公共服务等场景。现有研究大多围绕积分规则配置、商城交互设计、兑换路径优化和用户活跃提升展开[4][5]。随着数字化转型的深入，部分研究开始关注积分平台与推荐系统、运营活动和商家协同之间的联动关系，说明积分商城已逐渐从基础功能建设转向综合运营平台建设。

在工程实现方面，国内学者围绕微服务平台建设、行业业务系统重构、基于数据库表的服务拆分以及敏捷开发实践进行了较多探索[6][7][8][9][10][11][12][13]，也有研究从大数据营销视角讨论了个性化推荐与运营优化方法[14]。不过，从现有教学型项目和毕业设计成果看，仍普遍存在多角色协同不完整、关键业务一致性论述不足以及工程方案与开发成本不匹配等问题。本文围绕中国移动积分商城场景开展设计与实现，正是试图在业务完整性与工程可落地性之间取得平衡。

综合来看，当前相关研究已经为积分平台的功能设计和技术演化提供了较为丰富的参考，但在面向教学与实践结合场景的三端协同、业务一致性控制和平台治理能力方面仍有进一步研究空间。本文围绕中国移动积分商城场景开展设计与实现，正是试图在“业务完整性”和“工程可落地性”之间寻求平衡。

## 1.3 研究意义

本课题的研究意义主要体现在业务应用、工程实践和教学研究三个层面。

第一，业务应用意义。积分商城能够将沉淀积分转化为可感知、可消费、可追踪的权益资产，帮助平台打通“积分发放、积分消耗、订单转化、售后反馈、运营调整”的完整链路。通过引入多角色协同机制，平台不仅能够提升用户对积分价值的认知，还能增强商户供给与运营管理的联动效率。

第二，工程实践意义。本文以真实可运行的代码系统为基础，综合使用 FastAPI、Vue 3、SQLAlchemy、Redis、JWT 与 MySQL 等主流技术，完成从需求分析、架构设计到功能落地和测试验证的完整开发过程。研究过程中涉及统一鉴权、事务处理、缓存协同、软删除建模、接口分层和状态管理等典型工程问题，具有较强的综合实践价值。

第三，教学研究意义。相较于单一功能展示型项目，积分商城系统包含多角色权限控制、复杂业务状态流转和较为完整的数据实体体系，更能体现软件工程和信息系统开发的综合训练要求。本文形成的设计思路和实现经验，可为同类毕业设计课题以及教学型全栈系统开发提供参考样本。

# 第二章 系统相关技术概述

## 2.1 FastAPI 框架

FastAPI 是基于 Python 的现代 Web 开发框架，具有异步支持、类型提示友好、依赖注入清晰和接口文档自动生成等特点。与传统框架相比，FastAPI 在接口设计阶段更强调数据结构的显式声明与参数校验，这使得后端服务在多人协作和前后端联调场景中更具可维护性。

在本系统中，FastAPI 主要承担接口路由组织、请求参数校验、依赖注入和权限拦截等任务。系统通过路由分组的方式区分公开接口、用户接口、商户接口和管理员接口，并结合 Depends 机制实现身份校验与角色控制。与此同时，FastAPI 自带的 Swagger UI 与 ReDoc 文档能力降低了调试和联调成本，适合毕业设计项目在较短开发周期内实现接口工程化。

从适配性角度看，本课题需要同时处理认证、订单、商品、公告、活动等多个业务域，接口数量较多，且对开发效率和可读性要求较高。FastAPI 在性能与开发体验之间提供了较好的平衡，因此适合作为本系统的核心后端框架。

## 2.2 SQLAlchemy 与 MySQL 数据库

MySQL 作为成熟的关系型数据库，具有事务机制完善、生态丰富、部署维护成本较低等优势，适合承载积分商城中的用户、商品、订单、库存和积分流水等结构化数据。积分商城系统存在大量状态变化和多表关联场景，例如订单生成需要关联用户、商品、库存和积分记录，因此关系型数据库在数据约束和事务控制方面更具优势。

SQLAlchemy 2.x 是 Python 生态中常用的数据访问工具，既能够完成 ORM 映射，也支持较灵活的查询表达。结合本项目约束，系统未采用 relationship 进行隐式关联，而是通过业务层显式组织查询与联表操作。该方案虽然增加了一定代码量，但有助于开发者准确把握数据流向，降低隐式加载带来的调试复杂度，也更符合教学型项目强调“可读、可控、可验证”的实现原则。

此外，系统在数据库建模中统一引入 soft delete 设计，通过 deleted_at 字段保留逻辑删除痕迹，既满足后台管理与数据追踪需求，又避免物理删除导致的历史信息缺失。订单快照、积分流水和审核记录等设计，也体现了系统在数据可追溯性方面的考虑。

## 2.3 Redis 缓存技术

Redis 是高性能的内存型键值数据库，在缓存、分布式协调和高频状态控制等场景中应用广泛。对于积分商城系统而言，引入 Redis 可以降低部分高频状态对数据库的直接压力，并为会话控制、热点数据缓存和后续业务优化提供基础设施支持。

结合本系统的实际代码实现，Redis 主要承担运行支撑与认证扩展入口两类职责。其一，系统通过统一的 Redis 客户端和依赖注入机制，为 Token 校验链路预留黑名单检查能力，并在健康检查接口中验证缓存服务可用性。其二，Redis 作为系统的基础设施组件，为后续令牌撤销、热点缓存和高频业务状态控制提供扩展空间。

需要说明的是，当前版本的订单与库存一致性控制主要依赖 MySQL 事务和 inventory 表中的 locked_quantity 字段完成，Redis 并未直接参与下单事务中的库存扣减。因此，在本课题中 Redis 的定位更偏向运行支撑与后续优化接口，而数据库仍然是业务一致性与持久化存储的核心载体。

## 2.4 Vue 3、Pinia 与 Vite

Vue 3 是当前主流前端框架之一，具有响应式机制完善、组件化程度高和组合式 API 灵活等特点。积分商城系统涉及首页展示、商品详情、订单列表、后台表格和统计视图等多种交互形态，需要较强的组件拆分能力和状态响应能力，因此 Vue 3 能够较好满足本系统的前端开发需求。

Pinia 是 Vue 官方推荐的状态管理方案，具备结构简洁、类型友好和使用成本低等优点。在本系统中，Pinia 主要承担登录态、用户信息、购物车数量和积分余额等跨页面共享状态的维护工作。通过将共享状态统一存放于 Store 中，系统降低了组件层级之间重复传值的复杂度，也增强了三端页面在交互逻辑上的一致性。

Vite 作为现代化前端构建工具，开发启动速度快、热更新反馈及时、配置方式简洁，非常适合需要高频联调的前后端分离项目。系统前端在实际开发中借助 Vite 的代理能力完成接口转发，在提高开发效率的同时，也保持了工程结构的清晰性。

## 2.5 Element Plus 组件库

Element Plus 是面向 Vue 3 的桌面端组件库，提供表格、表单、弹窗、菜单、分页和通知等丰富组件。积分商城的商户端与管理员端包含大量典型后台页面，如商品管理、订单管理、商户审核、规则配置和报表展示等，这些页面对开发效率与界面一致性要求较高，采用成熟组件库能够显著缩短界面搭建周期。

在本系统中，Element Plus 主要用于表格化数据展示、弹窗式操作确认、分页查询、表单输入校验与布局组织。对用户端而言，其卡片、输入框、对话框和反馈组件也有助于提升页面交互统一性。虽然组件库会在一定程度上限制视觉个性化空间，但对于毕业设计项目而言，其优势主要体现在可快速形成规范界面、减少重复开发并提升交互稳定性。

## 2.6 JWT 认证机制与密码加密

前后端分离架构下，JWT 是较为常见的无状态认证方案。本系统采用 access token 与 refresh token 组合的会话管理方式，其中 access token 用于接口访问授权，refresh token 用于在访问令牌过期后刷新登录态。该机制既降低了频繁登录的操作成本，也能结合前端拦截器形成较平滑的会话续期体验。

在权限实现上，系统依据用户角色对访问范围进行约束。普通用户、商户与管理员分别访问不同路由空间，前端通过路由守卫进行页面拦截，后端再通过 Depends 依赖完成接口级身份校验。同时，系统在依赖层接入了基于 Redis 的令牌黑名单校验入口，为后续完善退出登录与令牌失效控制提供了基础。

在密码安全方面，系统使用 bcrypt 对用户密码进行哈希存储。相较于明文保存方式，哈希处理能够有效降低数据库泄露时的安全风险，是 Web 应用账号体系的基础安全要求。认证与加密机制的引入，为积分商城中的积分、订单与个人信息提供了必要的安全保障。

本系统围绕用户、商户和管理员三类主体设计功能入口，界面组织方式符合电商平台和后台管理系统的常见使用习惯。用户端以商品浏览、购物车和订单中心为主线，商户端以商品、库存和订单管理为主线，管理员端以审核、规则配置和平台运营为主线，功能路径相对清晰，学习成本较低。

本系统通过多种手段提高安全保障水平：在账号层面采用 bcrypt 进行密码哈希；在会话层面采用 JWT 双令牌机制并预留基于 Redis 的令牌校验扩展入口；在权限层面采用前后端协同拦截；在业务层面通过库存校验、事务控制和积分流水记录保证关键数据具备可追踪性。尽管本系统仍属于教学型实现，尚未接入真实短信网关、对象存储安全签名或支付风控服务，但在当前研究边界内已具备较完整的基础安全能力。

## 3.2 需求分析

### 3.2.1 用户端功能需求分析

用户端是积分商城价值实现的直接入口，其核心需求可概括为“持续获得积分、便捷消费积分、清晰追踪积分变化”三个方面。

在账号与个人中心方面，系统应支持用户注册、登录、登录态刷新、个人资料维护、密码修改和收货地址管理等功能。注册成功后需依据当前启用的注册积分规则赠送积分，个人中心还应展示积分余额、订单概览和签到状态，并支持每日签到获取额外积分。

在商品与消费体验方面，用户应能够浏览首页推荐内容、查看商品详情、按照分类或关键字进行检索，并在购物车中统一管理待兑换商品。除普通商品兑换外，系统还应提供话费与流量快捷充值入口，以及公告浏览和活动参与等功能，从而增强积分消费场景的丰富性。

在订单与售后管理方面，用户应能够查看订单列表、订单详情与状态变化，并完成取消订单、确认收货、提交评价和申请退款等操作。同时，系统还应提供积分流水、充值记录和退款处理状态查询功能，帮助用户形成对积分资产变化的完整认知。

### 3.2.2 商户端功能需求分析

商户端是平台供给能力和履约能力的主要承担者，其需求重点在于提升商品管理效率、库存可视化能力和订单处理效率。

首先，商户应支持入驻申请、资质提交和状态查询等基础功能，为平台审核和后续运营管理提供依据。系统需对商户设置 pending、approved、rejected 等审核状态，并保留驳回原因，便于追踪申请处理过程。

其次，商户应具备商品全生命周期管理能力，包括新增商品、编辑商品信息、维护 SKU、设置上下架状态、查看商品列表和管理库存数量等。由于积分商城中的可兑换商品直接关联库存消耗，系统还应提供可用库存计算与低库存预警能力，帮助商户及时补货。

最后，商户需要对所属订单进行查看、筛选和发货处理，并能够查看基础经营统计和活动参与情况。商户端的重点不在于平台治理，而在于高效完成商品供给、订单履约和经营数据查看等日常业务。

### 3.2.3 管理员端功能需求分析

管理员端承担平台治理和运营配置职责，其需求不仅是“查看数据”，更重要的是“维护规则”和“控制流程”。

在平台治理层面，管理员应能够查看用户与商户信息，对异常用户进行封禁或解封，对商户申请进行审核，并对退款申请、公告内容和活动状态进行统一管理。由于商户审核结果会直接影响平台运营秩序，系统必须对审核状态和处理原因进行留痕。

在运营配置层面，管理员应能够维护注册赠积分规则，发布或编辑公告，创建活动并审核商户报名商品。通过这些能力，平台可以在不修改核心代码的前提下完成一定程度的运营调整。

在分析决策层面，管理员还应能够查看平台概览、订单数据、待审核商户、退款处理结果和积分变化情况，为后续运营优化提供数据支持。因此，管理员端应体现“平台治理者”而非普通后台用户的角色定位。

### 3.2.4 安全性需求分析

系统涉及账号信息、积分资产、订单状态和库存数据，安全性是基础约束。系统应保证密码不以明文形式存储，访问令牌校验可靠，不同角色之间的数据访问边界清晰，并为敏感业务操作保留必要的状态记录与错误提示。

### 3.2.5 性能与可靠性需求分析

商品列表、订单列表和积分流水等高频查询接口应具备分页能力，避免大批量数据一次性加载造成响应延迟。下单、取消订单、确认收货和退款审核等场景需要在一定并发条件下保持结果正确，防止出现积分与订单、库存与订单状态不一致的问题。同时，系统还应具备基本的异常处理和统一响应能力，便于前端快速识别错误来源。

### 3.2.6 可维护与可扩展性需求分析

后端应遵循路由层、业务层与数据层职责分离的原则，前端应保持接口层、状态层和页面层边界清楚，以便后续定位问题和扩展功能。在完成核心积分商城功能的基础上，系统还应为消息通知、优惠券、推荐算法和更细粒度报表分析等模块预留扩展空间，避免因初期设计过于耦合而限制后续演进。

# 第四章 系统总体设计

## 4.1 总体设计

### 4.1.1 系统总体架构设计

本系统采用前后端分离架构，整体由前端表现层、后端业务层、数据库持久化层和缓存支撑层组成。前端基于 Vue 3、Vue Router、Pinia 与 Element Plus 构建用户端、商户端和管理员端三套界面；后端基于 FastAPI 提供统一 API 接口，并按 Router、Service、Model 分层组织认证、商品、订单、积分、活动和公告等业务；MySQL 负责核心数据持久化，Redis 为认证校验入口、健康检查和后续缓存优化提供基础设施支持。

从职责划分看，后端遵循 Router、Service、Model 的分层设计。Router 层负责参数解析、权限依赖与统一响应调用；Service 层集中处理业务逻辑、事务控制和跨表操作；Model 层负责定义实体结构与基础查询。该架构能够降低不同业务域之间的耦合度，使接口扩展、异常定位和代码维护更为清晰。

从调用链路看，前端通过统一请求模块访问后端接口，请求经后端权限依赖与业务服务处理后写入 MySQL，并以统一结构返回前端；Pinia 负责登录态与共享状态维护，Axios 拦截器负责 Token 注入与刷新重试。此种架构既保证了三端功能边界清楚，也为后续模块扩展提供了较稳定的基础。

## 4.2 系统功能流程设计

### 4.2.1 积分兑换下单流程设计

积分兑换下单是系统最核心的业务流程。用户在购物车或商品详情页选择 SKU 与数量并提交订单后，系统首先执行商品状态校验、SKU 合法性校验、可用库存校验、积分余额校验以及收货地址校验。库存判断以 inventory 表中的 quantity 与 locked_quantity 为依据，只有当 quantity 减去 locked_quantity 后仍满足购买数量时才允许继续下单。

在通过前置校验后，系统在同一数据库事务中完成订单主表写入、订单明细快照写入、用户积分扣减、积分流水新增、库存锁定量增加和商品销量更新等关键操作。当前系统采用纯积分即时扣减模式，因此订单创建后直接进入已支付状态。该流程的核心思想是以事务保证订单、积分和库存锁定三类数据同时成功或同时失败，从而避免产生半完成数据。

### 4.2.2 快捷充值流程设计

快捷充值流程主要服务于话费与流量两类轻量积分消费场景。用户在快捷充值页面选择手机号、充值类型和预置套餐后，系统首先校验手机号格式、充值类型和积分余额是否满足条件。校验通过后，系统按照预置兑换规则计算本次所需积分，并在数据库中同步完成积分扣减、积分流水新增和 phone_recharge_orders 记录写入。

该流程的设计重点在于将快捷充值视为独立于实物订单之外的积分消费链路，使其在不依赖物流履约的情况下仍然具备完整的积分追踪能力。通过独立记录充值类型、套餐名称和消耗积分，系统能够较好支持话费与流量两类轻量业务的统一管理。

### 4.2.3 订单履约与退款审核流程设计

在订单创建完成后，商户可在后台查看待处理订单并填写物流信息完成发货，系统据此将订单状态切换为已发货。用户确认收货后，订单进入已完成状态，同时释放对应的 locked_quantity，使锁定库存转化为实际成交结果，从而结束订单履约流程。

若用户对已支付、已发货或已完成订单发起退款申请，系统首先创建 refund_requests 记录，并将订单状态更新为 refunding。管理员在后台审核退款申请后，若审核通过，则更新退款申请状态、向用户返还积分并新增 refund 类型积分流水，同时将订单状态改为 refunded；若审核拒绝，则退款申请状态变为 rejected，订单恢复为 completed。当前版本的退款流程重点处理订单状态与积分资产闭环，未进一步扩展逆向入库管理。

## 4.3 数据库设计

### 4.3.1 概念结构设计

本系统数据库采用关系型结构设计，以用户、商户、商品、库存、订单和积分为核心对象构建数据模型，并扩展了签到、快捷充值、退款申请、公告和活动等辅助业务表。所有主要业务表均包含主键、创建时间、更新时间和软删除标记字段，以支持数据追踪和逻辑删除需求。

从实体划分看，用户实体用于保存平台账号主体信息，是认证、积分和订单模块的基础；商户实体用于支撑入驻与审核流程；商品、SKU 和库存实体用于表达商品展示信息、规格价格与库存状态；订单与订单明细实体用于记录兑换行为与收货信息快照；积分规则与积分流水实体用于描述积分来源、消费和返还过程；退款申请、签到和快捷充值实体则用于支撑扩展业务场景。

考虑到订单历史展示不能受商品或地址后续修改影响，系统在订单主表和订单明细表中保留了收货信息、商品名称、规格名称和图片等快照字段，提高历史订单展示的稳定性和可解释性。

# 第五章 系统实现

## 5.1 认证与用户端功能实现

### 5.1.1 登录认证与会话续期实现

认证与权限模块是系统运行的前置条件。后端认证接口提供发送短信验证码、用户注册、商户注册、登录、刷新 Token、退出登录、个人资料维护和修改密码等能力。其中短信验证码采用固定值进行演示，以减少教学环境对外部短信网关的依赖。

在注册流程中，系统先校验手机号唯一性并使用 bcrypt 对密码进行哈希存储，再根据积分规则表查询当前启用的注册赠积分规则，写入用户余额与 points_records 流水；登录成功后，系统生成 access token 和 refresh token，并将用户身份信息返回前端。前端通过 request.ts 中的 Axios 实例统一处理请求头注入和 401 自动刷新，auth Store 使用 Pinia 维护 token、refresh token 和 userInfo，并同步保存到 localStorage，从而保证页面刷新后的登录态延续。

需要说明的是，当前版本已在依赖层接入 Redis 黑名单校验入口，但退出登录接口仍以演示返回为主，完整的令牌注销写入逻辑仍可在后续继续完善。

### 5.1.2 商品浏览与积分兑换实现

商品模块负责连接前台消费体验与商户供给管理。系统采用分类、商品、SKU、商品图片分表设计：前台公开接口支持商品分类浏览、关键词检索、详情查看、SKU 展示和评价查看；用户可在商品详情页直接选择规格和数量，也可将商品加入购物车后统一结算。页面层通过分页、分类筛选与关键字检索降低一次性加载的数据量。

## 6.3 测试中发现的问题与改进方向

综合测试过程可以看出，系统核心业务链路已经能够稳定运行，但仍存在进一步完善空间。

第一，当前退出登录接口仍以演示返回为主，虽然依赖层已经具备 Redis 黑名单校验入口，但完整的令牌注销写入逻辑尚未完全闭环。

第二，当前测试仍以手工功能验证为主，自动化测试与性能压测覆盖度有待提高，对高并发场景下的容量上限尚缺少量化结论。

第三，系统尚未接入真实短信、支付和对象存储服务，部分业务环节仍采用模拟方式实现；对于逆向入库、更加细粒度的审核约束和更丰富的经营分析能力，也可以继续扩展。

## 6.4 测试结论

综合测试结果可以看出，本系统已能够较稳定地支持积分商城主要业务流程运行。用户端、商户端和管理员端的核心功能均可正常使用，订单、积分与库存锁定之间的关键数据能够保持基本一致，公告、活动、退款审核和积分规则等运营功能也可以较好支撑平台管理需求。就毕业设计目标而言，系统已达到较好的实现完整度和验证效果。

# 结 论

本文围绕中国移动积分商城场景，完成了一套面向用户、商户和管理员三类主体的积分商城系统设计与实现工作。研究从运营商积分业务的现实问题出发，针对传统积分平台在用户体验、商户协同、规则维护和数据一致性方面的不足，提出并落地了一种以前后端分离架构为基础、以三端协同为核心、以积分与订单一致性控制为重点的系统实现方案。

在系统设计层面，本文构建了覆盖认证鉴权、商品与库存管理、积分兑换、快捷充值、每日签到、订单履约、退款审核、商户审核、活动运营与公告管理的完整业务体系，并通过分层架构、软删除建模、统一响应封装和权限依赖控制提升系统可维护性。在关键业务处理层面，系统将积分扣减、订单创建和库存锁定纳入统一逻辑链，主要依靠数据库事务、订单快照和锁定库存字段保证状态一致性，从而提高了平台运行的稳定性和可解释性。

从实现与测试结果看，系统能够较好支撑积分商城核心业务闭环，具备一定的工程实践价值和教学示范意义。相比单一功能展示型项目，本文工作更强调多角色协作、规则驱动和数据可追踪性，因而更能体现信息系统开发中的综合设计能力。

同时，本文也存在一定局限。其一，系统尚未接入真实短信服务、支付接口与对象存储，部分业务环节仍采用模拟方式实现；其二，当前退出登录的令牌注销链路和更严格的商户准入约束仍可继续完善；其三，自动化测试、压力测试和异常恢复机制仍有增强空间。未来可在消息通知、优惠券体系、可观测性建设和真实外部服务对接等方向继续深入，以进一步提升系统的真实性、智能性与可落地性。
